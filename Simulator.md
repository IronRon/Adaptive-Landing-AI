# Simulator Guide

This document explains the simulator used to test and evaluate the contextual
slate bandit in this project.

## Purpose

The simulator is an offline testing harness that lets you:

- Generate synthetic visitor contexts.
- Use **real DB arms** (`BanditArm`) for slate decisions.
- Use your **real bandit selection** function (`choose_slate`).
- Use your **real update path** (`update_stats`) to train LinUCB params in DB.
- Compare against random and no-change baselines.
- Export reproducible artifacts (CSV + plots + summary stats).

This gives you a safe environment to validate behavior before or alongside live
traffic.

## Files Involved

- Command runner: `landing/management/commands/simulate_bandit.py`
- Simulator helpers: `landing/simulator.py`
- Bandit core logic (unchanged): `landing/bandit_utils.py`

## How To Run

Basic run:

```bash
python manage.py simulate_bandit --rounds 20000 --k 3 --epsilon 0.1 --seed 42
```

Useful options:

- `--reset-params`: reset learned DB params before simulation.
- `--dry-run`: do not update DB params (evaluation only).
- `--output-dir sim_outputs`: where CSV and PNG files are saved.
- `--ma-window 200`: moving-average smoothing window.

Examples:

```bash
# Fresh training run
python manage.py simulate_bandit --rounds 20000 --k 3 --epsilon 0.1 --seed 42 --reset-params

# Evaluation-only run (no learning updates)
python manage.py simulate_bandit --rounds 20000 --k 3 --epsilon 0.1 --seed 42 --dry-run
```

## High-Level Flow

Per round, the command does:

1. Pick a synthetic persona.
2. Build a synthetic context and feature vector.
3. Run bandit policy (`choose_slate`) and simulate reward.
4. If not dry-run, update real model params (`update_stats`) for chosen arms.
5. Run random baseline on same context (no updates).
6. Run no-change baseline on same context (no updates).
7. Store all 3 outcomes.

After all rounds:

1. Save CSV with all rows.
2. Build cumulative and moving-average curves.
3. Save two PNG plots.
4. Print summary CTR metrics.

## Policies Compared

- `bandit`: real contextual slate policy, can learn during run.
- `random`: random non-conflicting slate, no learning.
- `no_change`: control policy (`no_change` arm), no learning.

Random baseline uses the same conflict rules as production so comparison remains
fair and valid.

## Context Generation

Synthetic contexts are generated in `build_synthetic_context(...)`.

Includes:

- Persona type (price/trust/service/location/contact/balanced).
- Device (mobile/desktop).
- Intent scores in `[0, 1]` with noise and clipping.
- Visit count normalized to match bandit feature expectations.

Feature vector ordering is aligned with `FEATURE_NAMES` from bandit utils.

## Reward Model

Reward is generated by `simulate_reward(...)` and is binary:

- `reward = 1` means CTA click.
- `reward = 0` means no click.

A probability `p_click` is computed from simple rules:

- Base click rate.
- Bonus when chosen arms match persona focus topic.
- Penalty when focused persona is not matched.
- Balanced persona bonus for multi-topic coverage.
- Small intent-strength adjustment.
- Clipping to `[MIN_P_CLICK, MAX_P_CLICK]`.

Then reward is sampled from that probability.

All constants are centralized in `landing/simulator.py` with tuning ranges.

## Training Behavior

Training only occurs for bandit policy and only when not in dry-run:

- Command calls real `update_stats(arm, feature_vector, reward)`.
- `LinUCBParam` rows in DB are updated (A matrix, b vector, pull count).

Baselines are evaluation-only and do not modify model parameters.

## Outputs

### 1) CSV

Saved as:

- `simulate_bandit_seed<seed>_r<rounds>.csv`

Columns:

- `round`
- `policy`
- `persona`
- `device`
- `chosen_arm_ids`
- `p_click`
- `reward`

### 2) Plot: Cumulative Average

Saved as:

- `simulate_bandit_cumavg_seed<seed>_r<rounds>.png`

Shows long-run average reward (CTR) for each policy.

### 3) Plot: Moving Average

Saved as:

- `simulate_bandit_movingavg_seed<seed>_r<rounds>.png`

Shows smoothed local trend with configurable window (`--ma-window`).

### 4) Terminal Summary

Printed metrics include:

- Overall CTR per policy.
- CTR per persona per policy.
- CTR per device per policy.

## How To Evaluate Results

Use this checklist:

1. Bandit overall CTR should beat random and no-change consistently.
2. Bandit curves should trend above baselines over time.
3. Persona-level CTR should be reasonable, not only one persona winning.
4. Results should be stable across multiple seeds.

Recommended experiment pattern:

1. Run multiple seeds (for example 1, 2, 3, 4, 5).
2. Test multiple epsilon values (for example 0.05, 0.1, 0.2).
3. Compare final CTR and curve behavior across runs.

## Suggested Calibration Workflow

Current reward constants are heuristic by design. To calibrate:

1. Set `BASE_P_CLICK` close to your observed real baseline CTR.
2. Tune match bonus and mismatch penalty to reflect plausible uplift/drop.
3. Re-run across seeds and verify bandit > random is reliable.
4. Keep constants fixed while comparing algorithmic changes.

## Common Pitfalls

- Running without active arms in DB.
- Comparing one seed only (high variance risk).
- Using only dry-run and expecting model learning.
- Over-interpreting very short runs (too few rounds).

## Quick FAQ

Q: Does simulator change bandit core logic?
A: No. It imports and uses existing selection/update functions.

Q: Does random policy train model?
A: No. Only the bandit branch updates stats.

Q: Why include no-change baseline?
A: It is the control benchmark to verify that any adaptation is beneficial.
